### Requirement: Поля записи транзакции

Каждая транзакция SHALL содержать обязательные поля: наименование, сумма, дата, тег, категория (`Расход` / `Доход`), специфика (`Обязательный` / `Простой`); опциональные: комментарий, ссылка (URL), файл-вложение.

#### Scenario: Создание минимальной транзакции
- **WHEN** пользователь заполняет все обязательные поля и сохраняет
- **THEN** запись появляется в основной таблице транзакций

#### Scenario: Отклонение неполной транзакции
- **WHEN** пользователь пытается сохранить транзакцию без обязательного поля
- **THEN** система показывает ошибку валидации и не сохраняет запись

### Requirement: Интерфейс добавления транзакции

Добавление транзакции SHALL открываться через кнопку «+ Добавить» в боковой панели или модальном окне с быстрым переключением «Расход» / «Доход».

#### Scenario: Переключение категории в форме
- **WHEN** пользователь переключает категорию между «Расход» и «Доход» в форме
- **THEN** форма сохраняет введённые данные и меняет контекст категории

#### Scenario: Автодополнение наименования
- **WHEN** пользователь начинает вводить наименование
- **THEN** система предлагает варианты из ранее введённых наименований

#### Scenario: Сохранение черновика при закрытии
- **WHEN** пользователь закрывает форму с несохранёнными данными
- **THEN** черновик сохраняется локально и восстанавливается при следующем открытии формы

### Requirement: Фильтрация таблицы транзакций

Таблица SHALL поддерживать серверную фильтрацию по: наименованию (подстрока), сумме (диапазон), дате (диапазон), тегу/подтегу (мультиселект), категории, специфике.

#### Scenario: Фильтр по диапазону дат
- **WHEN** пользователь задаёт дату «от» и «до» и применяет фильтр
- **THEN** таблица показывает только транзакции в указанном диапазоне

#### Scenario: Фильтры в URL
- **WHEN** пользователь применяет один или несколько фильтров
- **THEN** состояние фильтров сериализуется в URL и восстанавливается при открытии ссылки

### Requirement: Действия над записью

Пользователь SHALL иметь возможность редактировать, удалять (с подтверждением) и дублировать транзакцию.

#### Scenario: Дублирование транзакции
- **WHEN** пользователь выбирает «Дублировать» для записи
- **THEN** открывается форма с предзаполненными полями (кроме даты — текущая или исходная по решению UX)

#### Scenario: Удаление с подтверждением
- **WHEN** пользователь инициирует удаление транзакции
- **THEN** система запрашивает подтверждение и после согласия удаляет запись

### Requirement: Вложения файлов

Транзакция MAY содержать одно вложение: JPG, PNG, WEBP, PDF, DOC/DOCX, максимум 10 МБ.

#### Scenario: Загрузка изображения
- **WHEN** пользователь прикрепляет JPG ≤10 МБ
- **THEN** файл сохраняется (локально `uploads/` или S3), в таблице отображается миниатюра-превью

#### Scenario: Отклонение слишком большого файла
- **WHEN** пользователь прикрепляет файл >10 МБ
- **THEN** система отклоняет загрузку с понятным сообщением об ошибке

#### Scenario: Замена и удаление вложения
- **WHEN** пользователь редактирует транзакцию и заменяет или удаляет файл
- **THEN** старый файл удаляется из хранилища, в БД обновляются `file_path`, `file_name`, `file_mime_type`

### Requirement: Ссылка в транзакции

Опциональное поле URL SHALL отображаться в таблице как кликабельная ссылка с иконкой, открывающаяся в новой вкладке.

#### Scenario: Отображение внешней ссылки
- **WHEN** у транзакции задан валидный URL
- **THEN** в таблице отображается иконка ссылки; клик открывает URL в новой вкладке

### Requirement: Пагинация и производительность

При количестве записей >500 таблица SHALL использовать пагинацию или виртуальный скролл; фильтрация выполняется на стороне сервера.

#### Scenario: Серверная пагинация
- **WHEN** в системе более 500 транзакций и пользователь открывает таблицу
- **THEN** загружается страница записей с сервера, а не весь набор целиком
