## ADDED Requirements

### Requirement: Загрузка CSV-файла

Страница `/import` SHALL предоставлять зону drag-and-drop и кнопку «Выбрать файл», принимающие только файлы с расширением `.csv`. После выбора файл SHALL отправляться на backend (`POST /import/batches`), где выполняется парсинг и сохранение строк в таблицу модерации; кодировка — UTF-8, разделитель — запятая, первая строка — заголовки.

#### Scenario: Успешная загрузка файла
- **WHEN** пользователь перетаскивает или выбирает `.csv` файл
- **THEN** файл отправляется на сервер, сервер парсит его и сохраняет строки в таблицу модерации; под зоной загрузки отображается имя файла и количество строк (например, «transactions_june.csv · 24 строки»)

#### Scenario: Отклонение файла неподходящего формата
- **WHEN** пользователь пытается загрузить файл не с расширением `.csv`
- **THEN** система отклоняет файл и не отправляет его на сервер

#### Scenario: Восстановление незавершённого импорта после перезагрузки страницы
- **WHEN** пользователь открывает страницу `/import`, а в системе уже есть незакрытый батч импорта
- **THEN** таблица модерации сразу заполняется строками этого батча, полученными с сервера

### Requirement: Справка по формату CSV

Рядом с кнопкой «Выбрать файл» SHALL отображаться иконка-вопрос; клик по ней SHALL открывать справку с описанием формата файла: обязательные колонки (`title`, `amount`, `date`), необязательные (`tag`, `category`, `specificity`, `comment`, `url`), требования к каждому полю (кодировка UTF-8, разделитель — запятая, формат даты `YYYY-MM-DD`, формат суммы — число с точкой, допустимые значения `category` — `expense`/`income`, допустимые значения `specificity` — `required`/`simple`, формат тега — путь через `/`) и пример строки CSV.

#### Scenario: Открытие справки по формату
- **WHEN** пользователь нажимает иконку-вопрос рядом с кнопкой «Выбрать файл»
- **THEN** открывается окно со списком обязательных/необязательных колонок, требованиями к формату каждого поля и примером строки CSV

#### Scenario: Закрытие справки
- **WHEN** пользователь закрывает окно справки (крестик, клик вне окна или Esc)
- **THEN** окно справки закрывается, зона загрузки остаётся доступной

### Requirement: Валидация строк при парсинге

Сервер SHALL присваивать строке статус `Ошибка`, если: отсутствует обязательное поле (`title`, `amount`, `date`); `amount` содержит нечисловое значение; `date` не соответствует формату `YYYY-MM-DD`; `category` не входит в `expense`/`income`; `tag` (с учётом иерархии через `/`) не найден в дереве тегов системы. Эта же проверка SHALL выполняться повторно на сервере после каждого inline-редактирования строки.

#### Scenario: Ошибка нечислового значения суммы
- **WHEN** значение колонки `amount` в строке — нечисловая строка (например, «тысяча»)
- **THEN** строка получает статус `Ошибка`, ячейка «Сумма» подсвечивается красной рамкой с tooltip-описанием ошибки

#### Scenario: Ошибка неверного формата даты
- **WHEN** значение колонки `date` не соответствует формату `YYYY-MM-DD` (например, «15.06.2026»)
- **THEN** строка получает статус `Ошибка`, ячейка «Дата» подсвечивается красной рамкой с tooltip-описанием ошибки

#### Scenario: Ошибка ненайденного тега
- **WHEN** значение колонки `tag` (включая путь через `/`, например «Еда/Кафе») не совпадает ни с одним тегом/подтегом в системе
- **THEN** строка получает статус `Ошибка`, ячейка «Тег» подсвечивается красной рамкой с tooltip-описанием ошибки

#### Scenario: Сводка ошибок над таблицей
- **WHEN** после парсинга найдены строки со статусом `Ошибка`
- **THEN** над таблицей модерации отображается список ошибок вида «Строка N: поле "<поле>" <описание> — "<значение>"»

### Requirement: Таблица модерации с inline-редактированием

После загрузки файла под зоной загрузки SHALL появляться таблица модерации со столбцами: чекбокс, Название, Сумма, Дата, Тег, Категория, Специфика, Комментарий, Ссылка, Статус, Действие; данные таблицы — строки текущего батча, полученные с сервера. Любая ячейка SHALL быть редактируема inline: клик превращает её в input/select, сохранение отправляет изменение на сервер (`PATCH /import/rows/{id}`); поле «Тег» использует тот же компонент дерева тегов, что и форма транзакции (включая inline-создание нового тега); «Категория» и «Специфика» — inline-select. Поля `title`, `amount`, `date` редактируемы, но не могут быть очищены до пустого значения.

#### Scenario: Inline-редактирование ячейки
- **WHEN** пользователь кликает на ячейку таблицы модерации
- **THEN** ячейка переключается в режим редактирования (input/select) с текущим значением

#### Scenario: Сохранение правки отправляется на сервер
- **WHEN** пользователь подтверждает новое значение ячейки
- **THEN** изменение отправляется на сервер, сервер пересчитывает статус и ошибки строки, обновлённая строка отражается в таблице

#### Scenario: Запрет очистки обязательного поля
- **WHEN** пользователь пытается сохранить пустое значение в ячейке «Название», «Сумма» или «Дата»
- **THEN** система не принимает пустое значение и сохраняет предыдущее

#### Scenario: Жёлтая рамка незаполненного поля принятия
- **WHEN** строка не содержит ошибок парсинга, но поле `tag`, `category` или `specificity` не заполнено
- **THEN** соответствующая ячейка подсвечивается жёлтой рамкой

### Requirement: Статусы строки и автоматический пересчёт

Строка модерации SHALL иметь один из статусов: `Ожидает` (нет ошибок, но не все обязательные для принятия поля заполнены), `Готово` (все поля заполнены, строка может быть принята), `Ошибка` (есть ошибки парсинга). Статус SHALL автоматически пересчитываться на сервере после каждого изменения ячейки. Статус `Перенесено` не хранится как состояние строки модерации — после принятия строка переносится в транзакции и удаляется из таблицы модерации (см. «Принятие строк в транзакции»); на фронте `Перенесено` — кратковременное визуальное состояние строки перед её исчезновением из списка.

#### Scenario: Исправление ошибки переводит строку в Готово
- **WHEN** пользователь исправляет все проблемные ячейки строки со статусом `Ошибка` и все поля принятия (`tag`, `category`, `specificity`) заполнены
- **THEN** статус строки автоматически меняется на `Готово`

#### Scenario: Исправление ошибки переводит строку в Ожидает
- **WHEN** пользователь исправляет ошибку парсинга строки, но поле `tag`, `category` или `specificity` остаётся незаполненным
- **THEN** статус строки автоматически меняется на `Ожидает`

### Requirement: Панель управления над таблицей

Над таблицей модерации SHALL отображаться чекбокс «Выбрать все готовые» (выбирает только строки со статусом `Готово`), счётчики «Всего / Готово / Ошибок / Перенесено» и кнопка «Принять выбранные (N)», активная только при N > 0. Счётчики SHALL обновляться в реальном времени при изменении строк.

#### Scenario: Выбор всех готовых строк
- **WHEN** пользователь отмечает чекбокс «Выбрать все готовые»
- **THEN** выделяются только строки со статусом `Готово`; строки `Ожидает`, `Ошибка`, `Перенесено` не выделяются

#### Scenario: Счётчик обновляется при изменении строки
- **WHEN** статус хотя бы одной строки меняется
- **THEN** счётчики «Всего / Готово / Ошибок / Перенесено» немедленно пересчитываются; «Перенесено» вычисляется как разница между исходным числом строк батча и числом строк, оставшихся в таблице модерации

### Requirement: Принятие строк в транзакции

Кнопка «Принять» в строке SHALL быть активна только при статусе `Готово`; при статусе `Ожидает` или `Ошибка` кнопка disabled, а tooltip перечисляет незаполненные/проблемные поля. Массовое принятие выделенных готовых строк SHALL запрашивать подтверждение («Добавить N транзакций в таблицу расходов?») перед отправкой. Принятие (`POST /import/rows/{id}/accept` или `POST /import/batches/{id}/accept`) SHALL атомарно создавать транзакцию(и) и удалять принятую строку(и) из таблицы модерации; баланс в хедере пересчитывается. На фронте строка перед исчезновением на 1–2 секунды визуально помечается как «Перенесено», затем убирается из активного списка.

#### Scenario: Disabled-кнопка с перечислением проблем
- **WHEN** строка имеет статус `Ожидает` с незаполненными полями «Тег» и «Категория»
- **THEN** кнопка «Принять» в строке disabled, tooltip показывает «Заполните: Тег, Категория»

#### Scenario: Одиночное принятие готовой строки
- **WHEN** пользователь нажимает «Принять» на строке со статусом `Готово`
- **THEN** создаётся транзакция, строка удаляется из таблицы модерации, баланс в хедере пересчитывается; на фронте строка на 1–2 секунды отображается как «Перенесено» перед исчезновением из списка

#### Scenario: Массовое принятие с подтверждением
- **WHEN** пользователь выделяет несколько готовых строк и нажимает «Принять выбранные (N)»
- **THEN** появляется диалог подтверждения «Добавить N транзакций в таблицу расходов?»; после подтверждения создаются N транзакций, соответствующие N строк удаляются из таблицы модерации, счётчик «Перенесено» увеличивается на N

### Requirement: Завершение импорта

Когда в таблице модерации не осталось строк со статусом `Ожидает` или `Готово` (оставшиеся строки — только `Ошибка`, остальные перенесены и удалены), над таблицей SHALL появляться баннер завершения импорта со сводкой («Импорт завершён: M транзакций добавлено, K строк с ошибками»), кнопкой «Перейти к транзакциям» (редирект на `/transactions`) и кнопкой «Загрузить новый файл» (закрывает батч, сброс страницы).

#### Scenario: Баннер завершения после переноса всех готовых строк
- **WHEN** в таблице модерации не осталось строк со статусом `Ожидает` или `Готово`
- **THEN** отображается баннер со сводкой и кнопками «Перейти к транзакциям» / «Загрузить новый файл»

#### Scenario: Сброс страницы для нового импорта
- **WHEN** пользователь нажимает «Загрузить новый файл»
- **THEN** текущий батч закрывается на сервере (`POST /import/batches/{id}/close`), таблица модерации, счётчики и баннер сбрасываются, отображается пустая зона загрузки
