# Proposal: refactor-testcase

## Проблема

Проект развивался итеративно — каждый change добавлял новый функционал, но не всегда было время на рефакторинг и тесты. В результате накопился технический долг:

- **Backend**: местами неоптимальные паттерны работы с ошибками, дублирование логики, отсутствующие edge-case проверки, неполное тестовое покрытие.
- **Frontend**: непоследовательные паттерны между фичами, потенциальные утечки, неоптимальное состояние.
- **Тест-кейсы**: отсутствует единый источник правды по сценариям для E2E автоматизации (Playwright).

## Цель

1. **Рефакторинг** — пройтись по всей кодовой базе, найти и исправить неоптимальный код, архитектурные проблемы и баги; покрыть изменения тестами.
2. **Тест-кейсы** — написать детальные Playwright-совместимые тест-кейсы на весь проект и сложить их в `testcases/`.

## Scope

### Часть 1: Refactor

- Backend: handler, service, repository, middleware, pkg
- Frontend: features, components, lib

### Часть 2: Test Cases (для Playwright)

Разделы:
- Auth (login, logout, session)
- Balance (отображение, редактирование)
- Transactions (CRUD, фильтрация, пагинация, файлы)
- Tags (CRUD, подтеги, цвета)
- Import (загрузка CSV, модерация, принятие)
- Mandatory Payments (CRUD, mark-paid, подсветка дат)
- Analytics (календарь расходов)
- Navigation & Layout

## Non-goals

- Полная переработка архитектуры
- Смена стека
- Новый функционал

## Capability

`refactor-testcase` v1
