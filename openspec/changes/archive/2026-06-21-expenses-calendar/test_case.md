# Тест-кейсы: expenses-calendar

**Окружение:** dev (Docker backend + Vite frontend)
**Данные:** seed-фикстуры транзакций за несколько месяцев (минимум текущий месяц + один предыдущий, с подтегами и без них), плюс отдельный сценарий с данными только в одном месяце для проверки скрытия «назад»

---

## 4. Hover-попап

### TC-08: Режим «По Статьям»

| Ожидаемый результат | Список отдельных транзакций дня (название, сумма), сумма строк = сумме за день |

## 6. REST API агрегации

<!-- spec: expenses-calendar -->

### TC-12: `GET .../expenses-calendar?level=day`

| Поле | Значение |
|------|----------|
| Приоритет | P0 |
| Тип | Positive |
| Спека | expenses-calendar → «Запрос месячного уровня» |
| Шаги | 1. `GET /api/v1/analytics/expenses-calendar?level=day&year=2026&month=6` с валидной сессией |
| Ожидаемый результат | `200`, `items` содержит запись на каждый день июня (28-31), включая дни с `amount=0, hasData=false` |
| Статус | ☐ |

### TC-13: `GET .../expenses-calendar?level=month`

| Поле | Значение |
|------|----------|
| Приоритет | P0 |
| Тип | Positive |
| Спека | expenses-calendar → «Запрос годового уровня» |
| Шаги | 1. `GET /api/v1/analytics/expenses-calendar?level=month&year=2026` |
| Ожидаемый результат | `200`, `items` содержит ровно 12 элементов, `breakdownByTag`/`transactions` пустые |
| Статус | ☐ |

### TC-14: Невалидный `level`

| Поле | Значение |
|------|----------|
| Приоритет | P1 |
| Тип | Negative |
| Спека | expenses-calendar → «Невалидный level» |
| Шаги | 1. `GET /api/v1/analytics/expenses-calendar?level=week&year=2026` |
| Ожидаемый результат | `422`, тело ошибки с понятным сообщением |
| Статус | ☐ |

### TC-15: `level=day` без `month`

| Поле | Значение |
