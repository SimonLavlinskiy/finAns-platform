## ADDED Requirements

### Requirement: Колонки таблицы транзакций

Главная таблица SHALL отображать колонки: Наименование (клик → деталь/редактирование), Сумма (цвет и знак ±), Дата (`DD.MM.YYYY`, сортировка по клику), Тег (breadcrumb `Родитель / Подтег`, клик → фильтр), Категория (бейдж Расход/Доход), Специфика (бейдж Обязательный/Простой), Вложение (иконка скрепки/ссылки), Действия (⋯ → Редактировать, Дублировать, Удалить).

#### Scenario: Отображение суммы расхода
- **WHEN** в таблице отображается транзакция категории «Расход»
- **THEN** сумма показана красным цветом со знаком `−` и выровнена по правому краю моноширинным шрифтом

#### Scenario: Клик по тегу фильтрует таблицу
- **WHEN** пользователь кликает breadcrumb тега в строке таблицы
- **THEN** таблица фильтруется по этому тегу, фильтр отражён в URL

#### Scenario: Относительная дата для недавних записей
- **WHEN** дата транзакции — сегодня, вчера или в пределах 7 дней
- **THEN** в колонке «Дата» отображается относительная подпись (`Сегодня`, `Вчера`, `N дней назад`) с абсолютной датой во всплывающей подсказке

### Requirement: Пагинация таблицы

Таблица SHALL поддерживать пагинацию с выбором 25 / 50 / 100 записей на странице; переключатель внизу таблицы.

#### Scenario: Смена размера страницы
- **WHEN** пользователь выбирает 50 записей на странице
- **THEN** запрос к API содержит `per_page=50`, таблица перезагружается с первой страницы

#### Scenario: Переход на вторую страницу
- **WHEN** пользователь нажимает «Следующая страница» при `total_pages > 1`
- **THEN** загружается `page=2` с сохранением активных фильтров

### Requirement: Пустое состояние таблицы

При отсутствии записей или результатов фильтрации SHALL отображаться иллюстрация «Нет записей» и кнопка «Добавить первую транзакцию».

#### Scenario: Пустая БД
- **WHEN** транзакций нет и фильтры не заданы
- **THEN** показывается empty state с CTA «Добавить первую транзакцию»

#### Scenario: Пустой результат фильтра
- **WHEN** фильтры заданы, но совпадений нет
- **THEN** показывается empty state с предложением сбросить фильтры

### Requirement: Drawer-форма транзакции

Добавление и редактирование SHALL выполняться в боковой панели (Sheet/Drawer) шириной ~480px справа; таблица остаётся видна под оверлеем (opacity ~40%).

#### Scenario: Переключатель Расход/Доход
- **WHEN** пользователь переключает категорию в форме
- **THEN** акцент формы меняется (красный/зелёный) и знак суммы подставляется автоматически

#### Scenario: Сохранить и добавить ещё
- **WHEN** пользователь нажимает «Сохранить и добавить ещё»
- **THEN** транзакция сохраняется, форма сбрасывается к значениям по умолчанию, панель остаётся открытой

#### Scenario: Отмена с несохранёнными данными
- **WHEN** пользователь закрывает форму с введёнными данными
- **THEN** показывается диалог «Отменить без сохранения?»; при подтверждении черновик сохраняется в `localStorage`

#### Scenario: Восстановление черновика
- **WHEN** пользователь открывает форму и в `localStorage` есть черновик
- **THEN** система предлагает «Восстановить черновик?»

### Requirement: Панель фильтров

Над таблицей SHALL располагаться сворачиваемая панель фильтров: наименование (debounce 300ms), сумма (от/до), дата (пресеты + произвольный диапазон), теги (tree multiselect), категория, специфика; активные фильтры — чипы с крестиком; кнопка «Сбросить всё».

#### Scenario: Фильтры в query params
- **WHEN** пользователь применяет фильтр `category=expense` и `date_from=2026-01-01`
- **THEN** URL содержит соответствующие query params и состояние восстанавливается при открытии ссылки

#### Scenario: Сброс пагинации при смене фильтра
- **WHEN** пользователь меняет любой фильтр
- **THEN** запрос отправляется с `page=1`

### Requirement: REST API транзакций

Backend SHALL предоставлять endpoints: `GET/POST /api/v1/transactions`, `GET/PUT/DELETE /api/v1/transactions/:id`, `POST /api/v1/transactions/:id/duplicate`, `GET /api/v1/transactions/suggestions?q=`, `POST/DELETE /api/v1/transactions/:id/file`; фильтрация через query params (`search`, `amount_min`, `amount_max`, `date_from`, `date_to`, `tag_ids`, `category`, `specificity`, `page`, `per_page`, `sort_by`, `sort_order`).

#### Scenario: Создание транзакции
- **WHEN** клиент отправляет валидный `POST /api/v1/transactions`
- **THEN** ответ `201 Created` с телом `{ "data": { ... } }` включая вложенный объект `tag`

#### Scenario: Список с пагинацией
- **WHEN** клиент запрашивает `GET /api/v1/transactions?page=2&per_page=25`
- **THEN** ответ содержит `data` и `meta: { page, per_page, total, total_pages }`

#### Scenario: Дублирование
- **WHEN** клиент вызывает `POST /api/v1/transactions/:id/duplicate`
- **THEN** создаётся копия с текущей датой, ответ `201`

#### Scenario: Валидация суммы
- **WHEN** клиент отправляет `amount <= 0`
- **THEN** ответ `422` с `error.fields.amount`

### Requirement: Суммы в копейках

API SHALL хранить и передавать суммы как целое число в копейках (`234500` = 2345.00 ₽); frontend форматирует отображение с пробелом-разделителем тысяч.

#### Scenario: Сохранение суммы
- **WHEN** пользователь вводит `2 345` в форме
- **THEN** в API уходит `amount: 234500`

### Requirement: Загрузка UI — skeleton

Таблица SHALL показывать skeleton-плейсхолдер при загрузке данных (не спиннер); загрузка файла — прогресс-бар в зоне drop.

#### Scenario: Skeleton при первой загрузке
- **WHEN** страница транзакций открыта и данные ещё не получены
- **THEN** отображается skeleton строк таблицы
