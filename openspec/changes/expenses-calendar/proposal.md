# expenses-calendar: Календарь расходов с drill-down (год → месяц → день)

## Зачем

Сейчас в приложении нет ни одного представления, где видно динамику расходов во времени — пользователь может оценить траты только построчно в общей таблице транзакций. Нужен компактный визуальный календарь расходов (бар-чарт по дням/месяцам с переходом между уровнями и разбивкой по тегам на день), который закрывает требование «Расходы по периодам» из capability `analytics` и даёт первый реальный экран аналитики в продукте.

## Что меняется

- Блок «Календарь расходов» на странице `/transactions` (над фильтрами и таблицей, без отдельной страницы): горизонтальный бар-чарт, два уровня детализации — **год** (12 столбцов-месяцев) и **месяц** (столбцы по дням месяца)
- Навигация между периодами: стрелки «← Месяц/Год →»; кнопка «назад» скрывается, если в более раннем периоде нет транзакций; переход «вперёд» всегда доступен
- Текущий день/месяц подсвечивается отдельным цветом (синий) на фоне остальных (серый) столбцов
- Дни/месяцы без расходов отображаются как минимальный плейсхолдер-столбик (не нулевая высота)
- Под графиком — подпись периода (например «Июнь 2026») и сумма расходов за период
- Hover/tap по столбцу дня показывает попап: сумма за день, дата, переключатель **«По Расходам» / «По Статьям»**:
  - «По Статьям» — стек-бар + список верхнеуровневых тегов (название, сумма, % от дня)
  - «По Расходам» — список отдельных транзакций за день (название, сумма)
- Клик по столбцу дня (вне попапа) переходит на `/transactions` с фильтром по этой дате
- Клик по столбцу месяца в годовом виде переключает календарь на месячный уровень за выбранный месяц
- Новый backend-эндпоинт агрегации расходов по дням/месяцам с разбивкой по верхнеуровневым тегам для попапа

## Non-цели

- Группировка по неделям и произвольный диапазон дат — остаются как заглушка для будущих итераций capability `analytics`
- Сравнение «доходы vs расходы» и круговая/treemap-диаграмма по тегам за период — отдельные delta-спеки `analytics` (v2, уже описаны в каноническом spec, не реализуются в этом change)
- Лимиты трат, уведомления о превышении бюджета — вне scope
- Drill-down на уровень «день → список транзакций» как отдельная страница/модалка — реализуется через переход на существующую `/transactions` с фильтром, без новой UI-сущности

## Capabilities

### Новые capabilities

- `expenses-calendar`: виджет-календарь расходов (бар-чарт год/месяц, навигация по периодам, hover-попап с разбивкой по тегам/транзакциям, переход в таблицу транзакций по клику на день)

### Изменяемые capabilities

- `analytics`: реализация требования «Расходы по периодам (v2)» для группировок «день» и «месяц» через виджет `expenses-calendar`, встроенный блоком на существующую страницу `/transactions`

## Влияние

- **finAns-backend** (`workspace/finAns-backend/`): новый `internal/{handler,service,repository}/analytics_*.go`; SQL-агрегация транзакций по `date_trunc('day'|'month', date)` и верхнеуровневому тегу; новый эндпоинт `GET /api/v1/analytics/expenses-calendar`
- **finAns-frontend** (`workspace/finAns-frontend/`): новая `src/features/analytics/` (компонент календаря `ExpensesCalendar` + попап `CalendarDayPopup`), встроенная блоком в `TransactionsPage`; без отдельного маршрута/пункта меню; без `recharts` — кастомный div-бар-чарт (см. design.md, отклонение D6)
- **finAns-platform:** delta-спеки → `openspec/specs/{expenses-calendar,analytics}/` после archive
- **API:** `GET /api/v1/analytics/expenses-calendar?level=day|month&year=&month=`
- **Миграции:** не требуются (агрегация по существующим таблицам `transactions`, `tags`); опциональный индекс по `transactions(date)` для производительности — см. design.md
