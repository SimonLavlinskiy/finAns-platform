## Context

Канонический `analytics` (`openspec/specs/analytics/spec.md`) уже декларирует требование «Расходы по периодам (v2)»: столбчатая диаграмма с группировкой по дням/неделям/месяцам и переключателем периода. До этого change страницы `/analytics` не существует — нужно реализовать первую конкретную визуализацию этого требования: календарь расходов с уровнями «год» и «месяц».

Модель данных не меняется: `transactions(date, amount, tag_id, category)`, `tags(id, parent_id, name, color)`. У каждой транзакции ровно один тег, возможно — подтег; для разбивки в попапе нужна агрегация до верхнеуровневого тега (см. D2).

## Goals / Non-Goals

**Goals:**

- Backend-эндпоint, отдающий готовую агрегацию по дням месяца или по месяцам года за один запрос (без N+1 на фронте)
- Frontend-виджет «Календарь расходов»: бар-чарт год/месяц, навигация по периодам, скрытие «назад» при отсутствии более ранних данных
- Hover-попап на день с переключателем «По Расходам» / «По Статьям» и разбивкой по верхнеуровневым тегам
- Клик по дню → переход в `/transactions` с фильтром по дате; клик по месяцу в годовом виде → переключение календаря на месячный уровень

**Non-Goals:**

- Группировка по неделям, произвольный диапазон дат — остаются нереализованными требованиями `analytics` (v2), не блокируют этот change
- Попап/разбивка по тегам для столбца **месяца** в годовом виде — по скриншотам не требуется
- Сравнение доходы/расходы, круговая/treemap-диаграмма по тегам — отдельные delta `analytics`
- Lazy-загрузка breakdown по дню отдельным запросом — в v1 отдаём весь месяц сразу (см. Risks)

## Decisions

### D1: Контракт эндпоинта агрегации

`GET /api/v1/analytics/expenses-calendar?level=day|month&year=YYYY&month=MM(для level=day)`

Ответ (`{"data": {...}}`, конвенция из `main-fin-table`):

```json
{
  "data": {
    "level": "day",
    "period": { "year": 2026, "month": 6 },
    "total": 11902000,
    "hasPrevious": false,
    "items": [
      {
        "key": "2026-06-01",
        "amount": 0,
        "hasData": false,
        "isCurrent": false,
        "breakdownByTag": [],
        "transactions": []
      },
      {
        "key": "2026-06-11",
        "amount": 436800,
        "hasData": true,
        "isCurrent": false,
        "breakdownByTag": [
          { "tagId": 3, "tagName": "Налоги", "color": "#7C3AED", "amount": 150000, "percent": 34.3 },
          { "tagId": 7, "tagName": "Хоз.расходы", "color": "#EAB308", "amount": 286800, "percent": 65.7 }
        ],
        "transactions": [
          { "id": 101, "title": "ЖКХ", "amount": 150000 },
          { "id": 102, "title": "Хозтовары", "amount": 286800 }
        ]
      }
    ]
  }
}
```

Для `level=month` структура та же, но `items[].key = "2026-01".."2026-12"`, `breakdownByTag`/`transactions` всегда пустые массивы (попап для месяца в годовом виде не нужен — см. Non-Goals), `period.month` отсутствует.

`amount` — int64 в копейках, как везде в проекте.

### D2: Свёртка подтегов до верхнеуровневого тега

Транзакция хранит `tag_id`, который может быть подтегом. Для `breakdownByTag` агрегация идёт по **корневому** тегу: на старте запроса строим map `tagID → rootTagID` через уже существующую логику дерева тегов (`tag_service` хранит `parent_id`-цепочки), затем `GROUP BY rootTagID` после джойна транзакций с этой map. Реализация — в Go после одного `SELECT` всех тегов (дешёво, тегов мало), а не рекурсивным SQL.

### D3: `hasPrevious` — скрытие кнопки «назад»

Вычисляется отдельным `EXISTS(SELECT 1 FROM transactions WHERE date < :period_start)` — дешёвый запрос с индексом по `date` (см. D7). `period_start` — первое число месяца (level=day) или 1 января года (level=month).

### D4: Плейсхолдер для дней/месяцев без расходов

`hasData=false` при `amount=0`. Frontend рисует минимальный фиксированный по высоте столбик (4px), серым, **без** hover-попапа — клик/hover на таких столбиках не открывает попап и не делает переход (наводить смысла нет, данных нет).

### D5: Подсветка текущего дня/месяца

`isCurrent` считается на backend через `time.Now()` в той же таймзоне, что уже используется для прочих дат в `transaction_service` (без новой TZ-логики — открытый вопрос ниже, если потребуется отдельная TZ).

### D6: Реализация бар-чарта — кастомные div-столбики, не Recharts

**Альтернатива A (Recharts `BarChart`)** — формально в роадмапе фронтенда заявлен Recharts, но макет — плоские прямоугольники без осей/грида/линий, и интерактивный попап с переключателем и списком тегов плохо ложится в модель `Tooltip` Recharts без избыточной кастомизации.

**Альтернатива B (выбрана): кастомные flex/div-столбики**, по аналогии с уже отработанным в проекте паттерном (`TagFilterPicker` — ручной dropdown без UI-библиотеки). Высота столбика — `% от max(amount) в текущем наборе items`. Попап — обычный позиционируемый React-компонент (`position: absolute` относительно контейнера календаря, calendar не открывается внутри `Sheet`/`Dialog`, поэтому риск повторения pointer-events-бага из предыдущего фикса ниже, но используем тот же паттерн позиционирования, что уже проверен в `TagFilterPicker`).

Решение пересматривается, если в следующих delta `analytics` понадобится multi-series график (доходы vs расходы) — там Recharts может быть оправданнее.

### D7: Индекс по `transactions(date)`

Добавляем `000006_add_transactions_date_index.up.sql`:

```sql
CREATE INDEX IF NOT EXISTS idx_transactions_date ON transactions(date);
```

Дёшево, ускоряет и текущую фильтрацию по диапазону дат, и агрегацию `GROUP BY date_trunc(...)`, и `EXISTS`-проверку из D3.

### D8: Маршрут и навигация

Новая страница `/analytics`, пункт меню «Аналитика» (иконка `BarChart3` из `lucide-react`) в `AppLayout` между «Метки» и «План».

### D9: TanStack Query

Key: `['analytics', 'expenses-calendar', level, year, month]`, `staleTime` по умолчанию (30s, как в `Providers`). Существующие mutation-хуки транзакций (`useTransactionMutations`) дополнительно инвалидируют `['analytics']` при create/update/delete — иначе календарь не обновится после правки транзакции.

## Risks / Trade-offs

| Риск | Митигация |
|------|-----------|
| Весь месяц транзакций (включая `transactions[]` на каждый день) в одном ответе — при аномально большом числе операций в месяц payload растёт | Для одного пользователя с обычным объёмом трат не критично; если станет проблемой — перейти на lazy-загрузку breakdown по дню отдельным запросом при hover |
| Свёртка подтегов в Go, а не в SQL — лишний обход дерева тегов на каждый запрос | Тегов и транзакций за месяц немного; при росте объёма — закэшировать map `tagID → rootTagID` в сервисе на TTL |
| Кастомные div-столбики вместо Recharts — больше ручного CSS/расчётов высоты | Оправдано простотой текущего макета; пересмотреть при появлении multi-series графиков |
| `EXISTS`-запрос для `hasPrevious` на каждый рендер периода | Один лёгкий индексный запрос, не проблема при текущем масштабе (один пользователь) |

## Migration Plan

1. `db/migrations/000006_add_transactions_date_index.up.sql` (+ `.down.sql` — `DROP INDEX`)
2. Backend: `internal/{repository,service,handler}/analytics_*.go`, роут `GET /api/v1/analytics/expenses-calendar` в защищённой группе (`RequireAuth`, как остальные `/api/v1/*` кроме `/health` и `/auth/*`)
3. Frontend: `src/features/analytics/` (страница, компонент `ExpensesCalendar`, hook `useExpensesCalendar`), маршрут `/analytics`, пункт меню в `AppLayout`, инвалидация `['analytics']` в существующих transaction-мутациях
4. Деплой backend → frontend (фронт зависит от нового эндпоинта)

## Open Questions

- **Таймзона для `isCurrent`**: использовать ли отдельную TZ-конвенцию (например, фиксированную `Europe/Moscow`) вместо текущей конвенции дат в `transaction_service`? → Рекомендация: не вводить новую логику в этом change, пересмотреть только если появится явная жалоба на «не тот день подсвечен»
- **Попап для месяца в годовом виде**: скриншоты не показывают hover на месяце — подтвердить с пользователем, что в этом change это действительно не нужно (сейчас зафиксировано как Non-Goal)
