## ADDED Requirements

### Requirement: Создание проекта
Система SHALL позволять аутентифицированному пользователю создать проект с полями `name` (обязательно), `started_at` (опционально), `initial_balance_kopecks` (опционально, по умолчанию 0). Создатель автоматически становится `owner`.

#### Scenario: Успешное создание проекта
- **WHEN** пользователь отправляет форму с `name="Семейный бюджет"`
- **THEN** создаётся `projects(id, name, owner_id, created_at)` и `project_members(project_id, user_id, role="owner")`; пользователь переходит в созданный проект

#### Scenario: Первый вход без проекта
- **WHEN** пользователь вошёл, но не состоит ни в одном проекте
- **THEN** отображается экран «У вас нет проектов» с кнопкой «Создать проект»

### Requirement: Переключение активного проекта
Если пользователь состоит в нескольких проектах, система SHALL предоставлять переключатель в хедере. Выбранный `project_id` хранится в `localStorage` и передаётся в каждом API-запросе через заголовок `X-Project-ID`.

#### Scenario: Переключение проекта
- **WHEN** пользователь выбирает другой проект в переключателе
- **THEN** `localStorage.activeProjectId` обновляется, все данные страницы перезагружаются для нового проекта

#### Scenario: Один проект — переключатель скрыт
- **WHEN** пользователь состоит ровно в одном проекте
- **THEN** переключатель проекта в хедере не отображается

### Requirement: Приглашение участника
Владелец проекта SHALL добавлять участников по `@username`. Участник добавляется мгновенно без подтверждения. Роль нового участника — `member`.

#### Scenario: Успешное приглашение
- **WHEN** владелец вводит `@anna_k` в разделе «Участники» и подтверждает
- **THEN** если пользователь `anna_k` существует — создаётся `project_members(project_id, user_id, role="member")`; участник появляется в списке

#### Scenario: Пользователь не найден
- **WHEN** владелец вводит `@unknown_user` которого нет в системе
- **THEN** возвращается ошибка `USER_NOT_FOUND`; запись не создаётся

#### Scenario: Участник уже в проекте
- **WHEN** владелец приглашает пользователя, который уже состоит в проекте
- **THEN** возвращается ошибка `ALREADY_MEMBER`

### Requirement: Роли и права участников
Система SHALL поддерживать две роли: `owner` (создатель) и `member` (равные права на CRUD данных). `member` не может удалять проект или других участников. `owner` не может покинуть проект, пока он единственный участник.

#### Scenario: Member пытается удалить участника
- **WHEN** пользователь с ролью `member` отправляет запрос на удаление другого участника
- **THEN** возвращается `403 FORBIDDEN`

#### Scenario: Owner пытается покинуть проект в одиночку
- **WHEN** единственный `owner` пытается удалить себя из проекта
- **THEN** возвращается ошибка `CANNOT_LEAVE_SOLE_OWNER`

### Requirement: Изоляция данных по проекту
Все сущности (транзакции, теги, лимиты, обязательные платежи, плановые расходы, баланс) SHALL быть привязаны к `project_id`. Запросы без валидного `X-Project-ID` из числа проектов пользователя SHALL возвращать `403 FORBIDDEN`.

#### Scenario: Запрос к данным чужого проекта
- **WHEN** пользователь отправляет запрос с `X-Project-ID` проекта, в котором не состоит
- **THEN** возвращается `403 FORBIDDEN`

#### Scenario: Запрос без X-Project-ID
- **WHEN** запрос к защищённому эндпоинту отправляется без заголовка `X-Project-ID`
- **THEN** возвращается `400 BAD_REQUEST` с кодом `PROJECT_ID_REQUIRED`

### Requirement: Настройки проекта
Система SHALL предоставлять страницу настроек проекта: название, список участников с ролями, форма добавления участника. Только `owner` видит кнопку удаления участника.

#### Scenario: Просмотр участников
- **WHEN** любой участник открывает настройки проекта
- **THEN** отображается список участников в формате `@username — роль`

#### Scenario: Удаление участника владельцем
- **WHEN** `owner` удаляет `member` из проекта
- **THEN** запись `project_members` удаляется; пользователь теряет доступ к данным проекта

## MODIFIED Requirements

### Requirement: Миграция существующих данных
При применении миграции SHALL быть создан дефолтный пользователь `@simon` + дефолтный проект `«Мои финансы»`, все существующие записи SHALL получить `project_id` этого проекта. Два существующих пользователя добавляются как участники дефолтного проекта.

#### Scenario: Применение миграции на непустой БД
- **WHEN** применяется миграция `000010_add_users_and_projects.up.sql` на БД с существующими данными
- **THEN** все строки в `transactions`, `tags`, `spending_limits`, `mandatory_payments`, `planned_expenses`, `user_balance` получают `project_id` дефолтного проекта; данные не теряются
